The one big idea

An agent is a node. A team is a graph.

A multi-agent system sounds like it needs a new engine. It does not. Each agent is just a function that reads the state and writes back to it — the exact definition of a Day-24 node. Wire a few of those nodes together and you have a team.

def researcher(state): ... # an agent = a NODE def writer(state): ... # an agent = a NODE def editor(state): ... # an agent = a NODE graph = StateGraph(State) # the team = a GRAPH graph.add_node("researcher", researcher) # ... wire them

No new library today. Multi-agent = Day 24's StateGraph with more nodes.

Shape 1 · Sequential

The assembly line: researcher → writer → editor

g = StateGraph(State) g.add_node("researcher", researcher) g.add_node("writer", writer) g.add_node("editor", editor) g.add_edge(START, "researcher") g.add_edge("researcher", "writer") # ordinary edges = a straight line g.add_edge("writer", "editor") g.add_edge("editor", END)

When to use: a fixed pipeline where each step needs the previous one's output.

Shape 1 · the hand-off

The hand-off lives in the state

Agents never talk directly. The researcher writes notes into the shared state; the writer reads notes and writes draft; the editor reads draft. The state is the conveyor belt.

class State(TypedDict): topic: str notes: str # researcher writes this draft: str # writer reads notes, writes this final: str # editor reads draft, writes this def writer(state): return {"draft": llm(f"Write about {state['notes']}")} # only the key it changed
Gotcha

A plain list key overwrites — last writer wins. If two nodes both append to one list, carry it forward yourself or use a reducer (Shape 3). For single string fields like above, overwrite is exactly what you want.

Shape 2 · Supervisor / hierarchical

A boss routes each turn to a worker

g.add_conditional_edges("supervisor", route, {"billing":"billing","tech":"tech","general":"general","done":END}) g.add_edge("billing", "supervisor") # every worker edges BACK to the boss g.add_edge("tech", "supervisor") # that back-edge makes it a hierarchy

When to use: the worker (or order) depends on the input — routing and judgement.

Shape 2 · the router

A word becomes the next node

The supervisor (an LLM or a rule) writes a decision word into the state. The router function — the same add_conditional_edges mechanic from Day 24 — turns that word into the name of the next node, or END.

def route(state) -> str: # supervisor put its choice here return state["next"] # "billing" # "tech" # "general" # or "done"
  • The router returns a key; the dict maps key → node name.
  • Workers edge back to the supervisor, so it can route again — a loop.
  • recursion_limit is the safety belt so a mis-routing boss can't spin forever.
Straight from Day 24

This is the conditional-edge + loop combo you already built — only now each destination is a whole agent instead of a plain function.

Shape 3 · Parallel (fan-out / fan-in)

Ask three specialists at once

for r in ["tone", "facts", "seo"]: g.add_edge("dispatch", r) # fan-out: one node -> several (run concurrently) g.add_edge(r, "combine") # fan-in : several -> one node (waits for all)

When to use: independent reviewers / opinions on the same input.

Shape 3 · the reducer

The reducer keeps every opinion

Three reviewers all write to the same field at the same time. Without help, the last one wins and two notes vanish. A reducer (Day 24) tells LangGraph to add updates instead of replacing them.

import operator from typing import Annotated class State(TypedDict): blurb: str # each reviewer appends one note notes: Annotated[list, operator.add] def tone_reviewer(state): return {"notes": ["tone: warmer intro"]}
  • Each specialist returns a single-item list.
  • Annotated[list, operator.add] concatenates them, so all three survive.
  • Plain notes: list → last writer overwrites, two notes lost.
Nothing new

Identical to Day 24's add_messages / operator.add reducers — just applied to a review panel.

Give the team a memory

Memory: exactly the Day-24 two knobs

from langgraph.checkpoint.memory \ import MemorySaver team = g.compile( checkpointer=MemorySaver()) cfg = {"configurable": {"thread_id": "roll-42"}} team.invoke({"topic": "UPI safety"}, cfg) team.invoke({"topic": "one more?"}, cfg)
  • Compile the whole team graph with a checkpointer.
  • A thread_id per conversation — same id remembers, new id starts clean.
  • The checkpointer saves the entire shared state, notes and drafts included.
No multi-agent magic

A team is one compiled graph. Memory works the same as for a single node — nothing special because there are many agents.

In practice

Real teams combine them — e.g. a supervisor that routes to a parallel review panel, whose output flows down a sequential polish line. You already have all three pieces.
